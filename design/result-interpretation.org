* Understanding ORGA Results: A Guide

This document explains how to interpret the JSON output generated by ORGA.

** 1. Categories (Classification)

The `categories` field represents the system's "best guess" at the institution's type.

- **Primary Category:** The first item in the list (e.g., `["Hospital", "Government"]` -> Primary is Hospital). This is the dominant identity detected.
- **Secondary Category:** Subsequent items. These reflect significant secondary attributes found in the text.
  - *Example:* A `Hospital` with a `Government` secondary tag often indicates a publicly funded or state-operated facility.
  - *Example:* A `University` with a `NonProfit` tag indicates its legal or operational status.

*Warning:* Do not treat secondary tags as errors. They are semantic features of the corpus.

** 2. Confidence Scores

The `confidence` fields (0.0 - 1.0) are **heuristic scores**, not statistical probabilities.

- **1.0 (Certain):** Extracted from structured data (JSON-LD) or highly specific meta-tags.
- **0.8 - 0.9 (High):** Strong regex matches in expected zones (e.g., a phone number in a footer).
- **0.5 - 0.7 (Medium):** Heuristic matches that passed basic validation but lacked strong context.
- **< 0.5 (Low):** Speculative matches. Often filtered out by default unless debug mode is on.

** 3. Debug Info

The `debug_info` block is the "Black Box Recorder". It explains *why* a decision was made.

- `classification_debug`: Shows the exact keywords matched, the scores for each category candidate, and the logic used to suppress weaker candidates.
- `filtered_social_links`: Lists links that looked like social media but were rejected (e.g., "Share this page" links).

** 4. Address "Raw" vs. Structured

ORGA prioritizes **capturing the address** over **perfectly structuring it**.
- The `raw` field is the ground truth. It is the literal text block extracted from the page.
- Fields like `street`, `city`, `postal_code` are best-effort extractions.
- If `raw` contains extra text (e.g., "Main Campus: 123 Main St"), this is a known limitation of non-LLM parsing. Trust `raw` for display; use structured fields for indexing.
